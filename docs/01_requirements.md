# 🧾 Keep Awake for VS Code 要件定義書

## 1. 🎯 背景と目的

VS Codeで長時間作業や処理を行う際、OSの自動スリープや画面OFFにより作業が中断される場合がある。

本拡張機能は、VS Code Desktop上でユーザーが明示的に有効化している間だけ、画面OFFおよび自動スリープを一時的に抑制することを目的とする。

OSの電源設定を変更せず、管理者権限を要求せず、VS Code拡張機能として軽量に利用できる構成とする。

## 2. 👥 利用者とステークホルダー

| 区分    | 役割                      | 期待価値                    |
| ----- | ----------------------- | ----------------------- |
| 利用者   | VS Code Desktopを利用する開発者 | 長時間作業中の画面OFFや自動スリープを防げる |
| 開発者   | 本拡張機能の実装・保守担当者          | OS別の差分を保守しやすい構成で実装できる   |
| 組織管理者 | 利用端末のポリシー管理者            | OS設定を変更しない拡張として扱える      |

## 3. 📚 用語・ドメイン定義

| 用語             | 定義                              | 備考                        |
| -------------- | ------------------------------- | ------------------------- |
| スリープ抑制         | OSのアイドル判定による自動スリープを一時的に防ぐこと     | 手動スリープは対象外                |
| 画面OFF抑制        | OSのアイドル判定によるディスプレイOFFを一時的に防ぐこと  | 画面ロックは対象外                 |
| 明示操作           | ユーザーによる手動スリープ、画面ロック、ノートPCの蓋閉じなど | 本拡張機能では抑制対象外              |
| Awake ON       | スリープ抑制機能が有効な状態                  | ステータスバーで表示する              |
| Awake OFF      | スリープ抑制機能が無効な状態                  | 起動時のデフォルト状態               |
| Awake ERROR    | スリープ抑制の開始に失敗した状態                | エラー通知を表示する                |
| Extension Host | VS Code拡張機能を実行するVS Code内の実行環境   | 利用者によるNode.jsの別途インストールは不要 |
| Codicon        | VS Code標準のProduct Icon          | ステータスバーの状態表示に利用する         |

## 4. 🎭 ユースケース / ユーザーストーリー

* 利用者は、VS Codeで長時間作業を開始する前に、ステータスバーのアイコンボタンをクリックしてスリープ抑制を有効化できる。
* 利用者は、スリープ抑制が不要になった時点で、同じアイコンボタンをクリックして無効化できる。
* 利用者は、現在の状態をステータスバーのCodiconと表示テキストから判断できる。
* 利用者は、スリープ抑制の開始に失敗した場合、ステータスバー表示とエラー通知から失敗を認識できる。
* 利用者は、VS Codeを終了した場合、スリープ抑制が自動的に解除されることを期待できる。

## 5. ✅ 機能要件

### 5.1 🎛 ステータスバー表示

* VS Codeのステータスバーに、スリープ抑制状態を示すアイコンボタンを表示する。
* ステータスバー表示は、状態に応じて以下の内容に切り替える。

| 状態       | 表示                      | 意味           | クリック時の動作    |
| -------- | ----------------------- | ------------ | ----------- |
| OFF      | `$(eye-closed) Awake`   | スリープ抑制なし     | スリープ抑制を開始する |
| STARTING | `$(loading~spin) Awake` | スリープ抑制の開始処理中 | 追加操作は受け付けない |
| ON       | `$(eye) Awake`          | スリープ抑制中      | スリープ抑制を解除する |
| ERROR    | `$(warning) Awake`      | スリープ抑制の開始失敗  | 再試行する       |

* 表示にはVS Code標準のCodiconを利用する。
* 色だけに依存せず、アイコン形状とテキストで状態を識別できるようにする。

### 5.2 🔄 ON/OFF切替

* ユーザーがステータスバーアイコンボタンをクリックすると、現在状態に応じてON/OFFを切り替える。
* OFF状態でクリックされた場合、スリープ抑制を開始する。
* ON状態でクリックされた場合、スリープ抑制を解除する。
* STARTING状態では、重複実行を防ぐため追加クリックを無視する。
* ERROR状態でクリックされた場合、スリープ抑制の開始を再試行する。

### 5.3 🟢 スリープ抑制開始

* 対象OSに応じた一時的なスリープ抑制処理を開始する。
* 抑制範囲は、画面OFFおよびOSのアイドル判定による自動スリープとする。
* OSの電源設定、ユーザー設定、管理者設定は変更しない。
* スリープ抑制の開始に成功した場合、状態をONに変更する。
* スリープ抑制の開始に失敗した場合、状態をERRORに変更し、エラー通知を表示する。

### 5.4 🔴 スリープ抑制解除

* ON状態でユーザーがクリックした場合、スリープ抑制を解除する。
* VS Codeの拡張機能が終了または無効化される場合、保持中のスリープ抑制を解除する。
* 解除後は状態をOFFに変更する。
* 解除処理は、OS設定を変更せず、起動中の抑制処理のみを終了する。

### 5.5 🖥 OS別対応

対象OSは以下とする。

| OS      | 実装方針                                      | 備考              |
| ------- | ----------------------------------------- | --------------- |
| Windows | `SetThreadExecutionState` を利用する           | 管理者権限は要求しない     |
| macOS   | `caffeinate -d -i` を利用する                  | OS標準コマンドを一時実行する |
| Linux   | `systemd-inhibit --what=idle:sleep` を利用する | systemd環境を前提とする |

* 対象外OSでは、有効化時にERROR状態へ遷移し、未対応OSであることを通知する。
* Linuxでは環境差が大きいため、`systemd-inhibit` が利用できない場合はERROR状態へ遷移する。

### 5.6 🚫 対象外操作

以下の操作は、本拡張機能の抑制対象外とする。

* ユーザーによる手動スリープ
* 画面ロック
* ノートPCの蓋閉じ
* 電源ボタン操作
* 組織ポリシー、MDM、EDRなどによる制限
* OSまたはハードウェアの強制的な省電力制御

### 5.7 🔔 エラー通知

* スリープ抑制の開始に失敗した場合、ステータスバーを `$(warning) Awake` に変更する。
* 失敗時はVS Codeのエラー通知を表示する。
* エラー通知には、可能な範囲で原因を含める。
* エラー通知は、組織ポリシーやOS制限の回避を促す内容にしない。

### 5.8 ⚙️ コマンド

以下のコマンドを提供する。

| コマンド                  | 内容                   |
| --------------------- | -------------------- |
| `Keep Awake: Toggle`  | 現在状態に応じてON/OFFを切り替える |
| `Keep Awake: Enable`  | スリープ抑制を開始する          |
| `Keep Awake: Disable` | スリープ抑制を解除する          |

* コマンドパレットから実行できるようにする。
* ステータスバークリック時は `Keep Awake: Toggle` を実行する。

## 6. 🛡️ 非機能要件

### 6.1 🎨 デザイン

* VS Code標準UIに馴染む軽量な表示とする。
* 独自WebView、サイドバー、専用設定画面はMVPでは実装しない。
* ステータスバー表示にはCodiconを利用し、独自画像アイコンは使用しない。
* ダークテーマ、ライトテーマ、Product Icon Themeに自然に追従できる構成とする。

### 6.2 🧭 ユーザビリティ

* ユーザーは1クリックでON/OFFを切り替えられる。
* 現在状態は、ステータスバー上で常に確認できる。
* 起動時のデフォルト状態はOFFとする。
* 自動的に前回ON状態を復元しない。
* 通知はエラー時を中心とし、通常操作では過剰に通知しない。

### 6.3 🔐 セキュリティ

* 管理者権限を要求しない。
* OSの電源設定を永続変更しない。
* 組織ポリシーやOS制限を回避する処理を行わない。
* 任意コマンド実行機能は提供しない。
* 外部ネットワーク通信は行わない。
* 収集・送信する利用者データを持たない。

### 6.4 ⚡ パフォーマンス

* 拡張機能の起動時に重い処理を行わない。
* スリープ抑制OFF状態では、常駐処理やポーリングを行わない。
* ON状態では、OS別の抑制処理を維持するために必要な最小限のプロセスまたはAPI呼び出しのみを行う。
* STARTING状態では、重複起動を防止する。

### 6.5 🧱 保守性

* OS別の処理は `SleepInhibitor` 相当の責務として分離する。
* UI状態管理とOS別抑制処理を分離する。
* OS別の失敗理由を追跡しやすいよう、エラー処理を共通化する。
* READMEに動作条件、制限事項、対象外操作を明記する。

## 7. 🔒 制約条件

* 対象環境はVS Code Desktopとする。
* VS Code for Web、github.dev、vscode.devは対象外とする。
* 利用者PCにNode.jsの別途インストールを要求しない。
* VS Codeに含まれるExtension Host上で拡張機能を実行する。
* Remote SSH、WSL、Dev Containers利用時も、ローカル端末のスリープ抑制を目的とするため、拡張機能はUI側で動作させる。
* `package.json` では `extensionKind: ["ui"]` を指定する。
* 管理者権限を要求しない。
* OS設定の永続変更は行わない。
* OS標準機能またはOS標準コマンドが利用できない環境では、機能を有効化できない場合がある。
* 組織ポリシー、MDM、EDRなどによる制限は回避しない。

## 8. ⚠️ リスクと課題

| リスク                                 | 影響                     | 対応方針                    |
| ----------------------------------- | ---------------------- | ----------------------- |
| 組織ポリシーによりOS標準コマンドやAPI呼び出しが制限される     | スリープ抑制を開始できない          | ERROR状態とエラー通知で明示する      |
| Linux環境で`systemd-inhibit`が存在しない     | Linuxで有効化に失敗する         | ERROR状態とエラー通知で明示する      |
| Windows実装方式によってはPowerShell制限の影響を受ける | Windowsで有効化に失敗する可能性がある | 可能であればPowerShell依存を避ける  |
| ユーザーが蓋閉じや手動スリープにも効くと誤解する            | 期待値と実挙動がずれる            | READMEとエラー文言で対象外操作を明記する |
| VS Code終了時の解除漏れ                     | スリープ抑制が残る可能性がある        | `deactivate` 処理で必ず解除する  |

## 9. ✅ 受け入れ条件

* VS Code Desktopで拡張機能を有効化すると、ステータスバーに `$(eye-closed) Awake` が表示される。
* VS Code起動直後の状態はOFFである。
* OFF状態でステータスバーをクリックすると、STARTINGを経てON状態になる。
* ON状態では、画面OFFおよびOSのアイドル判定による自動スリープが抑制される。
* ON状態でステータスバーをクリックすると、スリープ抑制が解除されOFF状態になる。
* スリープ抑制の開始に失敗した場合、`$(warning) Awake` が表示され、エラー通知が表示される。
* VS Code終了時または拡張機能無効化時に、スリープ抑制が解除される。
* 手動スリープ、画面ロック、蓋閉じは抑制対象外として扱われる。
* OS設定は変更されない。
* 利用者PCにNode.jsの別途インストールを要求しない。
